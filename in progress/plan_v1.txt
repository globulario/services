  HERE'S WHERE IT BREAKS — The Race

            │  ⑩ Multiple releases hit RESOLVED in the SAME tick:   │
            │                            │                           │
            │  "release/dns":   RESOLVED → reconcileResolved()      │
            │    hasAnyActivePlan? → NO (event succeeded, slot free) │
            │    stampAndDispatchPlan(dns)                            │
            │                     ┌──────┴───────────────┐           │
            │                     │ current = dns plan    │           │
            │                     │ status = PLAN_PENDING │           │
            │                     │ gen = 2               │           │
            │                     └──────┬───────────────┘           │
            │                            │                           │
            │  "release/rbac":  RESOLVED → reconcileResolved()      │
            │    hasAnyActivePlan? → YES (dns PENDING)    │           │
            │    *** GOOD: skips ✅                        │           │
            │                            │                           │
            │  "release/mcp":   RESOLVED → reconcileResolved()      │
            │    hasAnyActivePlan? → YES (dns PENDING)    │           │

            │                            │                           │
            │  ... dns executes, succeeds, gen=2 ...                 │

  What Actually Happened (the bug)

            │  ⑫ BUT — reconcileResolved has a subtle problem:      │
            │                            │                           │
            │  When reconcileResolved dispatches a plan, it patches  │
            │  the release to APPLYING with the plan_id. Then the   │
            │  reconcile loop comes back and runs reconcileApplying. │
            │                            │                           │
            │  reconcileApplying calls checkNodePlanStatuses() which │
            │  reads the plan STATUS from etcd — but the status key  │
            │  is SHARED across all plans (one status per node).     │
            │                            │                           │
            │  Timeline of the actual crash:                         │
            │                            │                           │
            │  T=0  dns dispatched, gen=2, plan_id=aaa               │
            │  T=1  node agent starts executing dns plan             │
            │  T=2  reconcile tick: mcp is RESOLVED                  │
            │        hasAnyActivePlan → YES (dns RUNNING)            │
            │        skips ✅                                         │
            │  T=3  dns plan SUCCEEDS, status=SUCCEEDED gen=2        │
            │  T=4  reconcile tick:                                   │
            │        dns: APPLYING → checkNodePlanStatuses           │
            │          status.State=SUCCEEDED → dns goes AVAILABLE ✅ │
            │        mcp: APPLYING → reconcileResolved               │
            │          hasAnyActivePlan → NO                          │
            │          dispatch mcp plan, gen=3, plan_id=bbb         │
            │          status overwritten: PLAN_PENDING gen=3        │
            │                            │                           │
            │  ⑬ PROBLEM: 10 releases hit RESOLVED in same tick     │
            │     They ALL see hasAnyActivePlan=NO because the       │
            │     status write from stampAndDispatchPlan hasn't       │
            │     propagated to the other goroutine iterations yet   │
            │                            │                           │
            │     WAIT — actually the work queue is single-goroutine │
            │     So they execute SEQUENTIALLY within the worker.    │
            │     After mcp dispatches (gen=3), the next release     │
            │     should see PLAN_PENDING gen=3.                     │
            │                            │                           │
            │  ⑭ Let me re-examine. The actual problem:             │
            │                            │                           │
            │  The releases that got stuck at APPLYING had           │
            │  plan_id=ee0952fd (mcp). But the current plan in etcd │
            │  is ca683142 gen=10. That means 10 plans were          │
            │  dispatched sequentially (gen 1→10), each overwriting  │
            │  the previous.                                         │
            │                            │                           │
            │  HOW? Because reconcileApplying checks:                │
            │    - Is the node's plan_id matching what I dispatched? │
            │    - NO! The plan_id in the status is from a LATER     │
            │      release that overwrote the slot.                  │
            │                            │                           │
            │  ⑮ The ACTUAL sequence:                                │
            │                            │                           │
            │  reconcileResolved runs for release A:                 │
            │    dispatch plan_id=aaa gen=1                           │
            │    release A status: APPLYING, node.plan_id=aaa        │
            │                            │                           │
            │  reconcileResolved runs for release B:                 │
            │    hasAnyActivePlan? → YES (aaa is PENDING)            │
            │    node error = "waiting for active plan"              │
            │    release B status: APPLYING (no plan_id)             │
            │                            │                           │
            │  node agent executes plan aaa, SUCCEEDS                │
            │                            │                           │
            │  Next tick — release A: APPLYING → reconcileApplying   │

            │                            │                           │
            │  Next tick — release B: APPLYING → reconcileApplying   │
            │    ❓ B has no plan_id (it was "waiting")               │
            │    checkNodePlanStatuses reads status gen=1 SUCCEEDED   │
            │    But B's node.plan_id is EMPTY — does it match?      │
