# Awareness — agent navigation index

You are about to edit Globular code. Awareness is the queryable knowledge
layer that tells you which invariants the code protects, which fixes are
forbidden, and which tests pin the contract. The full documentation lives
under [`docs/awareness/`](../awareness/) — this page is just a one-screen
map so you can find the right file fast.

If you've never used awareness on this codebase, read the linked docs in
the order below. If you're here mid-session, skim the table.

---

## Quick path

```bash
# Before editing high-risk code:
globular awareness briefing --file <path>          # always start here
globular awareness impact   --file <path>          # if briefing was thin

# Before a multi-step task with no specific file:
globular awareness preflight --task "<what you're about to do>"
```

MCP equivalents: `mcp__awg__awareness_briefing`,
`mcp__awg__awareness_impact`, `mcp__awg__awareness_preflight`,
`mcp__awg__awareness_resolve`, `mcp__awg__awareness_query`,
`mcp__awg__awareness_diagnose`.

## Authoring vocabulary at a glance

When you fix something worth remembering, anchor it. The node classes
and where they live:

| Class | Where to author | Notes |
|-------|-----------------|-------|
| Invariant | `docs/awareness/invariants.yaml` | A property that must hold; stable, rarely revised |
| FailureMode | `docs/awareness/failure_modes.yaml` | Known break with symptoms / root_cause / forbidden_fixes / required_tests |
| ForbiddenFix | `docs/awareness/forbidden_fixes.yaml` (+ `forbidden_fixes/`) | Phase B: standalone entries don't surface in queries yet; reference by ID from failure modes to make it visible |
| Test (required_test) | Inline in invariant/failure_mode entries or `docs/awareness/required_tests.yaml` | Link by exact test function name |
| Intent | `docs/intent/<id>.yaml` — one file per intent | Fields: `level`, `title`, `intent`, `agent_guidance`, `bad_smells`, `activation_triggers`, `expressed_by` |
| IncidentPattern | `docs/awareness/incident_patterns.yaml` | Recurring shape across multiple incidents |
| CodeSymbol / SourceFile | Generated by `annotation-scanner` from `@awareness` comments | Do not hand-author |

### `@awareness` annotation keys (enforced by scanner)

```
namespace, component, symbol, file_role,
implements, enforces, protects, relates_to,
calls, emits, reads, writes,
tested_by, forbidden_fix,
risk (low|medium|high|critical), owner
```

Reference-keys (value must be a class-qualified ID):
`implements, enforces, protects, relates_to, forbidden_fix`.

Source of truth: `awareness-graph/golang/scanner/scanner.go` (`AllowedKeys`).

## Building and reloading the graph

The single canonical build command — never run yaml2nt by hand:

```bash
cd /home/dave/Documents/github.com/globulario/awareness-graph
SERVICES_REPO=/home/dave/Documents/github.com/globulario/services \
  bash scripts/build-awareness-graph.sh
```

What it does:
1. Builds `annotation-scanner` and `yaml2nt`
2. Scans all annotated Go source trees → regenerates
   `docs/awareness/generated/*.yaml`
3. Compiles authored YAML + intent + generated YAML into N-Triples
4. Writes to `awareness-graph/golang/server/embeddata/awareness.nt`

To push the rebuilt seed to a live Oxigraph (port 7878):

```bash
curl -X PUT "http://localhost:7878/store?default" \
  -H "Content-Type: application/n-triples" \
  --data-binary @awareness-graph/golang/server/embeddata/awareness.nt
# expect HTTP 204
```

A healthy graph is ~21K triples emitted, ~15K unique after dedup, ~4 MB.
Materially less suggests the annotation-scanner step was skipped (someone
used an older recipe).

`--check` mode (`scripts/build-awareness-graph.sh --check`) regenerates
into a temp dir and exits non-zero if anything committed is stale.

## What's NOT in the graph yet (Phase B)

yaml2nt logs these as `known_unsupported`. They're authored but invisible
to `briefing` today:

- `activation_rules.yaml`, `status_tracker.yaml`, `learning_rules.yaml`,
  `namespaces.yaml`
- `forbidden_fixes.yaml`, `forbidden_assumptions.yaml`, `fix_cases.yaml`,
  `forbidden_fixes/*.yaml`
- `audit_suppressions.yaml`, `authority_rules.yaml`, `context_aliases.yaml`,
  `detector_mapping.yaml`
- `failuregraph_seeds/*.yaml`
- `knowledge/*.yaml`
- `generated/*_annotation_report.yaml`
- `docs/intent/meta/*.yaml`

Practical consequence: a `ForbiddenFix` entry is only visible to agents
today via the failure_modes that reference it. Author the failure_mode
link at the same time.

## Hook carve-outs (so you don't waste a briefing call)

The PreToolUse briefing hook denies edits to high-risk dirs without a
prior `briefing(file=)` call. It deliberately does NOT fire on these
narrative subfolders (briefing returns empty for them anyway):

```
docs/awareness/reports/
docs/awareness/decisions/
docs/intent/reports/
docs/intent/decisions/
```

Hook source: `.claude/hooks/enforce-briefing.sh`.

## Final response footer (required for non-trivial code tasks)

```
Awareness used:
- briefing:                <files / tasks called>
- impact:                  <files called, or "not needed">
- resolved IDs:            <ids expanded, or "none">
- invariants touched:      <ids the change supports or risks violating>
- failure modes considered:<ids in scope, including ones ruled out>
- forbidden fixes avoided: <tempting shortcut you didn't take>
- required tests:          <tests this change must satisfy>
- tests run:               <actual results>
- remaining uncertainty:   <what you don't know>
```

If awareness was unavailable: `Awareness status: DEGRADED — fallback evidence: …`.
If awareness was empty: `Awareness status: EMPTY — code/docs checked manually: …`.

Full version + rationale: [`awareness/claude_agent_footer.md`](../awareness/claude_agent_footer.md).
