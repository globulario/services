# Versioning Policy

## Semantic Versioning

All Globular services follow [Semantic Versioning 2.0.0](https://semver.org/):

- **PATCH** (0.1.x): Bug fixes, no API changes
- **MINOR** (0.x.0): New features, backward compatible
- **MAJOR** (x.0.0): Breaking changes

## Mono-Version Track

All Globular Go services share a single version track. When any service gets a version bump, all services are bumped together. This ensures:

- Consistent deployment across the cluster
- No version matrix compatibility issues
- Simple operator mental model: "the cluster is at version X"

**Exceptions**: Infrastructure packages (etcd, prometheus, alertmanager, minio, xds) keep their upstream versions.

## Version Allocation

Versions are allocated by the repository service via `AllocateUpload`:

```bash
# Automatic allocation (recommended)
globular deploy echo --bump patch
globular pkg publish --file pkg.tgz --repository addr --bump minor

# Explicit version (deprecated, shows warning)
globular deploy echo --version 0.1.0
```

The repository enforces:
- **Monotonicity**: New versions must be >= latest PUBLISHED version
- **Uniqueness**: Each version+platform combination gets one build_id
- **Reservation**: 5-minute TTL prevents concurrent version collisions

## Build Identity

- **build_id** (UUIDv7): The sole authoritative artifact identity. Generated by the repository on upload.
- **build_number**: Display-only monotonic counter. Never used for convergence or decisions.
- **version**: Human-facing semver label. Used for desired-state and operator communication.

## 1.0.0 Criteria

All of the following must be true before declaring 1.0.0:

1. All invariants (INV-1 through INV-10) tested automatically on every commit
2. Zero anomalies on a 3-node cluster for 7 consecutive days
3. All Phase 1-7 deployed and validated on live cluster
4. Containerized test cluster operational for CI
5. Operator course updated to match current implementation
6. At least one external operator has followed the course successfully
