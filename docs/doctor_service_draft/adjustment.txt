Adjustments I strongly recommend
1) Inventory completeness invariant is too weak / can be wrong

Claude proposes:

“Inventory.Components empty OR Inventory.Units empty” ⇒ incomplete 

implementation_plane

That’s risky. A node could legitimately report zero “components” depending on how you model them, or a partial inventory may still have non-empty arrays. You already discussed (earlier in your project) an explicit inventory_complete flag in status reporting.

Fix: Use an explicit inventory_complete boolean from NodeAgent/Controller status if available. If it’s not available in the NodeAgent inventory today, add it (or derive it from a clearly-defined rule like “agent has scanned systemd + packages and stamped inventory_complete=true”).

Outcome: fewer bogus “incomplete” reports.

2) “unit_files_present” depends on exact state strings (“not-found”)

This line is brittle:

UnitStatus.State == "not-found" 

implementation_plane

State strings vary (systemd, your agent mapping, etc.). If your agent already normalizes unit states to enums, prefer that. If it’s strings today, define a canonical set in one place and use it everywhere.

Fix: Create a small normalization function (e.g., NormalizeUnitState(s string) UnitStateEnum) and have invariants rely on the enum.

3) Drift mapping “hash mismatch => CONFIG_HASH_MISMATCH” is semantically off

Plan says:

DesiredServicesHash != AppliedServicesHash → CONFIG_HASH_MISMATCH (services) 

implementation_plane

That might be true, but it’s not necessarily “config hash”; it’s desired vs applied state hash. It could represent version changes, unit enablement, config, etc.

Fix: Rename drift categories or add a category:

DESIRED_APPLIED_HASH_MISMATCH (or STATE_HASH_MISMATCH)
Then later you can refine into “config/version/unit” when you have more structured data.

This avoids misleading diagnostics.

4) Node “reachable” threshold should be configurable

Hardcoding > 2 min for stale heartbeat is okay for dev, but in prod you’ll want it configurable.

Fix: move thresholds into config:

node_heartbeat_stale_seconds

maybe per-role overrides later

5) ExplainFinding needs a stable “finding_id”

The plan uses finding_id but doesn’t define how it’s generated. If it changes every request, the UI “ExplainFinding” will be annoying.

Fix: Make finding_id deterministic from content, e.g.:
hash(invariant_id + entity_ref + key evidence fields)
(Stable for the same snapshot; optionally include snapshot_id if you want it snapshot-scoped.)

6) “Audit events on each report request” conflicts slightly with “read-only”

Emitting audit events is a write (to EventService), but not a cluster mutation. Still, for “v1 read-only purity,” I’d make it optional.

Fix: add config flag:

emit_audit_events: true|false default false for v1, or default true if you’re comfortable.

7) Port selection and routing

Plan proposes default port 12500 and “check existing allocations.” 

implementation_plane


That’s fine, but I’d prefer: do not pick a hard-coded port until you confirm the allocation table in your repo.

Fix: make port required in config (or allocate from your existing port registry mechanism, if you have one).

Minor improvements (nice-to-have)

The separate clusterdoctor_client/clusterdoctor_client.go isn’t necessary for v1 unless you already follow that convention everywhere.

Collector deadlines: 10s is okay, but consider separate timeouts for list vs per-node calls (per-node calls should be shorter + parallelism capped).

Cache TTL 5s is good; ensure “singleflight” is implemented properly to avoid stampedes (Claude mentions it, so just make sure it’s real).
