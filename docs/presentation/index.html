01

The Problem

Modern distributed systems are powerful,
but hard to reason about.
  • Opaque execution — something fails, and you piece together the story from logs across 5 systems
  • Hidden state — config files, env vars, secrets, database flags scattered everywhere
  • Container tax — runtime, registry, overlay network, CNI plugins, storage drivers — before you even run your code
  • Implicit convergence — controllers guess, retry silently, and hope for the best
What if a cluster behaved exactly
the way you describe it —
explicitly, deterministically,
and transparently?
02

What is Globular

An open-source microservices platform for building and operating self-hosted distributed applications.

Native Binaries

Services are compiled Go binaries running under systemd. No containers, no runtime overhead.

Workflow-Driven

Every operation is a formal workflow with phases, failure classification, and full audit trail.

Single Source of Truth

All configuration, state, and service discovery lives in etcd. No env vars. No config files to sync.

AI-Ready

Explicit state makes AI reasoning possible. AI observes, diagnoses, and acts — through the same workflows you use.

03

The 4-Layer State Model

Every package is tracked across four independent truth layers.

1 Artifact Does this version exist and is it valid?
2 Desired What version should be running?
3 Installed What is actually on each node?
4 Runtime Is it running and healthy right now?

Drift in any layer triggers investigation. No silent failures. No "almost correct."

Classified Failures

CONFIG • PACKAGE • DEPENDENCY • NETWORK • REPOSITORY • SYSTEMD • VALIDATION

Each type has its own retry strategy. Network errors retry with backoff. Validation errors stop immediately.

Concurrency Control

Semaphore — max 3 concurrent workflows
Circuit breaker — pauses dispatch when unhealthy
Backoff — 5-minute cooldown on failure

No cascading storms. Predictable under stress.

Three Actors, Clear Responsibilities

🎯
Cluster Controller
Decides what should happen.
Watches desired state.
Dispatches workflows.
⚙
Workflow Service
Orchestrates how it happens.
Executes steps.
Tracks results.
🖥
​
Node Agent
Performs the work.
Manages systemd units.
Reports status.

No overlap. No ambiguity. Each part has a single responsibility.

06

Globular vs. Kubernetes

Aspect Kubernetes Globular
Deployment unit Container image (OCI) Native binary (.tgz)
Process supervisor kubelet + container runtime systemd
State management etcd (hidden, internal) etcd (user-facing API)
Failure handling Controller retry loops Classified failure workflows
Audit trail Events (ephemeral) Full workflow history
Configuration ConfigMaps, Secrets, env vars etcd (single source of truth)
Minimum footprint Control plane + workers + runtime Binary + etcd + systemd

09

What Globular Is Not

✕ Not a Kubernetes replacement for teams already running containerized workloads
✕ Not for polyglot apps with complex dependency trees (Python, Node.js, Java) — containers solve that better
✕ Not a cloud service — you own the machines, you run the platform
✕ Not magic — it's explicit, deterministic, and transparent. What you see is what you get.

Globular is the right choice when you deploy compiled binaries, want full control over the OS, and need platform-level coordination without the container abstraction.

10

How It Looks

# Bootstrap your first node
globular cluster bootstrap \
  --node localhost:11000 \
  --domain mycluster.local \
  --profile core --profile gateway

# Deploy a service
globular services desired set postgresql 0.0.3 \
  --publisher core@globular.io

# See what happened
globular workflow list --limit 5
RUN-ID SERVICE STATUS DURATION
wf-a3f21 postgresql SUCCEEDED 12.4s

# Check everything is aligned
globular services repair --dry-run
All services converged. No repairs needed.

When Things Break

You don't guess. You follow the system's own reasoning.
